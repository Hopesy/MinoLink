# 会话管理首屏历史加载优化设计

## 背景

当前 `/sessions` 页面在 `OnInitialized()` 内同步执行：

- `ClaudeNativeSession.GetAllProjects()`
- `CodexNativeSession.GetAllProjects()`

这两条路径都会做全量目录扫描，并且对每个 `jsonl` 会话文件继续读取首条用户消息/自定义标题作为摘要。历史会话量一大时，首屏渲染前会发生大量磁盘 I/O 与 JSON 解析，导致页面首次打开明显卡顿。

## 目标

1. 页面首次进入时先完成首屏渲染，不再被历史会话扫描阻塞。
2. 项目卡片列表只依赖轻量元数据：Agent 类型、工作目录、会话 ID、最后活跃时间。
3. 会话摘要改为按项目展开时再加载，避免为折叠状态的会话做无效读取。
4. 同一进程内复用会话目录快照，避免管理页多次进入反复全量扫描。
5. 不改变 `/resume`、`/switch` 等命令当前依赖完整摘要的行为。

## 方案对比

### 方案 A：只把页面改成异步加载

- 优点：改动小。
- 缺点：虽然不卡首屏线程，但后台仍会一次性读取所有摘要，磁盘压力不变；首次真正可用时间依旧偏长。

### 方案 B：元数据先行 + 摘要懒加载 + 进程内缓存（推荐）

- 优点：同时解决“首屏阻塞”和“无意义全量摘要读取”两个问题；二次进入页面几乎可直接复用缓存。
- 缺点：需要新增一个目录快照服务，并让页面维护项目级加载状态。

### 方案 C：应用启动时预热全部摘要

- 优点：页面打开时可能更快。
- 缺点：把卡顿从页面首开转移到应用启动；摘要总量大时依旧浪费 I/O。

结论：采用方案 B。

## 设计

### 1. 原生会话读取拆成“轻量列表”和“完整详情”两条路径

保留 `GetSessions(workDir)` 作为完整读取接口，继续供 `/resume` 等命令使用。

新增“仅元数据”读取能力用于管理页列表：

- Claude：枚举项目目录与 `jsonl` 文件，只取文件名、最后修改时间，不读取首条消息。
- Codex：仍需扫描 `session_meta` 以识别 `cwd` 与 `id`，但不再继续读取后续用户消息摘要。

这样首屏列表仍能正确排序和分组，但不会为每个会话预读摘要。

### 2. 新增 `NativeSessionCatalogService`

服务职责：

- 合并 Claude/Codex 两类轻量项目列表；
- 在内存中缓存最近一次列表快照；
- 支持 `forceRefresh` 强制重扫；
- 按 `(agentType, workDir)` 单独加载完整会话详情，用于卡片展开时补全摘要。

这样页面逻辑不再直接依赖静态扫描入口，也避免每次进入页面都重新扫盘。

### 3. `/sessions` 页面改为非阻塞加载

页面初始化改成：

- 先同步计算活跃会话横幅；
- 立即读取目录快照缓存，如果已有缓存则直接渲染；
- 再后台触发异步刷新；
- 若还没有缓存，则先显示加载态，等异步任务完成后刷新页面。

卡片展开时：

- 先展开容器；
- 若该项目的摘要尚未加载，则显示项目级 loading 文案；
- 后台拉取完整会话列表并替换当前项目数据；
- 已加载过的项目不重复读取。

### 4. 状态边界

页面需要新增以下状态：

- `_isLoadingProjects`：首屏/刷新中的列表加载状态；
- `_projectLoadErrors`：项目级详情加载失败提示；
- `SessionProjectGroup.SummariesLoaded`：该项目是否已补全摘要；
- `SessionProjectGroup.IsLoadingDetails`：该项目是否正在懒加载详情。

## 测试策略

1. 为 `NativeSessionCatalogService` 增加单元测试：
   - 首次加载会合并 provider 结果；
   - 第二次无强刷时直接复用缓存；
   - 详情加载只调用目标 provider 的完整读取接口。
2. 手工验证管理页：
   - 首次进入能立即看到 loading 或缓存页面，不再整页卡死；
   - 卡片展开后摘要会补齐；
   - 刷新按钮可重新扫描目录。
3. 最终运行 `dotnet test` 与 `dotnet build`。
